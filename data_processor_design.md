## Vision
현재의 단방향 크롤링 시스템을 넘어, 수집된 첨부파일(PDF, HWP, DOCX)과 원문 데이터를 LLM을 통해 100% 정확하게 구조화된 정형 데이터로 가공하는 파이프라인을 구축합니다.
Fundy 프로젝트(`D:\fundy_Project\fundy`)와 완전히 분리된 모듈형(Microservice-like) 아키텍처를 지향하며, 오직 "데이터 정제 및 구조화"에만 집중하여 0%의 실패 확률(Robustness)을 달성합니다.

### 10x Check
단순히 텍스트를 긁어오는 수준을 넘어, 지원금 문서의 복잡한 표 포맷 붕괴, API Rate Limit, 네트워크 단절 등 모든 예외 상황을 방어하는 무결성 프로세스를 만듭니다. 가공이 끝난 DB 파일(`fundy_records.db`) 하나만 넘겨주면, 프론트엔드 서비스(Fundy)의 동기화 스크립트(`sync_crawler_db.js`)가 아무런 텍스트 전처리 고민 없이 즉시 Supabase로 동기화할 수 있는 완벽한 상태의 데이터를 제공합니다.

## Accepted Scope (추진 핵심 목표)
- [x] **DB 필터링 체계 강화**: SQLite DB(`fundy_records.db`)를 읽어 아직 AI 가공이 되지 않은 미처리 공고만 필터링.
- [x] **다운로드 및 멀티모달 전처리**: 첨부파일(PDF, HWP)을 임시 다운로드하고, 페이지 단위 이미지로 변환(PDF/HWP -> PNG/JPEG)하는 로직 구축.
- [x] **LLM 연동 및 스키마 확정**: Gemini 2.5 Flash 멀티모달 API 연동 및 pydantic JSON Schema를 통한 100% 확정된 형태의 데이터 구조화.
- [x] **강력한 오류 복구 체계**: LLM API Timeout, Rate Limit 발생 시 지수 백오프(Exponential Backoff) 기반의 안전한 재시도(Retry) 로직.
- [x] **최종 DB 병합**: 추출된 데이터를 `processed_funding_records`에 저장할 뿐만 아니라, Fundy Sync 봇이 읽을 수 있도록 `funding_records`의 해당 컬럼들(details, targets 등)을 최종 갱신.

## System Architecture (Modular Pipeline)
```text
[ 1. Crawler Project ] (현재 디렉토리)
  └─ spider.py (크롤링 수행)
       ↓
  └─ funding_records.db (Raw 데이터 적재)
       ↓
  └─ data_processor_agent.py (★우리가 구현할 핵심)
       ├─ 파일 다운로드 & 이미지 변환
       ├─ Gemini 2.5 Flash (멀티모달 JSON 추출)
       └─ DB 데이터 클렌징 및 정형화 (Update)

=================== (물리적 파일 복사 또는 공유 볼륨 전달) ===================

[ 2. Fundy Service Project ] (D:\fundy_Project\fundy)
  └─ sync_crawler_db.js (★기존 스크립트 재사용)
       ├─ 가공된 funding_records.db 읽기
       └─ Supabase (PostgreSQL)에 깔끔하게 Upsert
```
